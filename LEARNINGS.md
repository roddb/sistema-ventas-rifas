### 2026-05-05 — Sesión 5 (Fase 5.A en main + Fase 5.B en feature branch)

- **2026-05-05** Convención técnica — En Tailwind 3.4+, **preferir `min-h-dvh` sobre `min-h-screen`** para layouts mobile-first. `100vh` en iPhone Safari incluye la barra de URL; cuando colapsa (al scrollear), el viewport "saltita" 60-90px y el contenido se reacomoda. `100dvh` (dynamic viewport height) tracks el viewport actualmente visible, sin jiggle. Tailwind 3.4 lo soporta nativo. _(Destino: solo registro)_ _(Origen: nit del final reviewer Fase 5.A; aplicado en Task 10 de Fase 5.B)_

- **2026-05-05** Convención técnica — **`scrollbar-hide` no es utility default de Tailwind 3.4**. Si lo usás en una clase sin declararlo, Tailwind silently lo descarta (no genera CSS) y el scrollbar sigue visible. Para strips horizontales de tabs (caso `RangeTabs`), declarar manualmente en `globals.css`: `.scrollbar-hide::-webkit-scrollbar { display: none; }` + `scrollbar-width: none`. Existe el plugin `tailwind-scrollbar-hide` pero declaración manual = 5 líneas vs 1 dependency más. _(Destino: solo registro)_ _(Origen: descubierto al implementar RangeTabs en Fase 5.B Task 4)_

- **2026-05-05** Herramienta — **Worktrees nested dentro del repo padre** (caso `.worktrees/fase-5b`) sufren conflicto de eslint: `next lint` walk-up detecta dos `.eslintrc.json` (padre + worktree) y emite "Plugin @next/next was conflicted". Fix: agregar `"root": true` al eslintrc del worktree. Eso detiene el walk-up de eslint en el worktree, sin afectar al padre. La opción alternativa (worktrees fuera del repo, ej en `~/.config/superpowers/worktrees/`) evita el problema pero sacrifica la familiaridad de "todo dentro del proyecto". _(Destino: CLAUDE.md como regla)_ _(Origen: setup worktree Fase 5.B)_

- **2026-05-05** Convención técnica — Cuando se reemplaza un componente productivo monolítico (ej: `RifasApp.tsx` de 1.587 líneas) por una arquitectura de componentes pequeños, **NO mantener el archivo legacy en coexistencia con el nuevo** (anti-patrón "RifasApp.v2.tsx"). Riesgo principal mitigado por feature branch en worktree: si algo sale mal, `git worktree remove` y volvés a main intacto. _(Destino: solo registro)_ _(Origen: estrategia migración Fase 5.B)_

- **2026-05-05** Convención de proceso — **Skeleton-then-meat strategy** en plans de implementación con muchas pantallas: Task 1 crea el container/shell con state + routing + placeholders (`<div>{step} placeholder</div>`); Tasks 2-N reemplazan cada placeholder por su pantalla real. - **2026-05-05** Convención técnica — **`<form className="contents">` en componentes wizard React**: cuando el screen necesita usar un `<form>` para `onSubmit` pero el layout (header + body scrolleable + sticky bottom) viene del `PageContainer` flex padre, `display: contents` hace que el form no rompa el flex layout. - **2026-05-05** Convención técnica — **Naming de tokens semánticos vs. literal del spec**: el spec listó tokens como `primary`/`text-primary`/`surface-elevated`; en la implementación se renombraron a `brand`/`ink`/`surface-raised`. ### 2026-05-04/05 — Sesión 4 (fix BUG-010 + spec rediseño UI)

- **2026-05-04** Convención técnica — El bloque `env` en `next.config.js` **fuerza a Next.js a inlinear el valor en build time** en TODO el bundle (incluso server code), pisando el comportamiento normal de runtime para variables `NEXT_PUBLIC_*`. Si la env var no está disponible en build (ej: en `RUN npm run build` del Dockerfile), Next.js inlinea el fallback del `||` y queda hardcoded forever en el JS compilado, ignorando lo que después setees en runtime. **Regla**: NO usar el bloque `env` en `next.config.js` para variables que pueden ser undefined en build time. Para variables server-only que dependen del entorno (URLs públicas, secrets), no usar prefijo `NEXT_PUBLIC_` (que es para variables que SÍ se exponen al cliente) y no incluirlas en el bloque `env`. _(Destino: CLAUDE.md como regla)_ _(Origen: BUG-010)_

- **2026-05-04** Convención de proceso — **Smoke tests automatizados de pago deben inspeccionar las URLs internas de la preference creada vía MP API**, no sólo verificar que `/api/preference` devolvió `200 + initPoint`. Cuando un buyer recibe `Algo salió mal · Código CPT01-{random}` al hacer click en "Pagar", **NO significa "tu cuenta MP está mal"** automáticamente. Las causas posibles incluyen: config inválida del preference (URLs malformed, vacías, con localhost), tipo de cuenta no habilitado, restricción del payer (ej: seller=buyer, no se puede pagar a uno mismo). - **2026-05-04** Técnico — El SDK `mercadopago@2.0.15` rechaza preferences con `auto_return: 'approved' as const` con error `auto_return invalid. back_url.success must be defined`, **incluso cuando back_urls tiene los 3 campos como URLs HTTPS válidas**. - **2026-05-04** Convención de proceso — **Smoke E2E parcial sin user action es valioso**: con curl + Turso MCP + secret de Secret Manager, se puede validar 80% del flujo de pago sin gastar plata real ni requerir interacción humana. Cubre: creación de purchase, creación de preference (válida access token + URLs), webhook con firma inválida (regression BUG-008-D), webhook con firma válida + paymentId ficticio (valida cadena HMAC + retry 503). - **2026-05-04** Técnico — `openssl dgst -sha256 -hmac SECRET -hex | awk '{print $2}'` puede salir vacío si openssl falla a stderr. - **2026-05-04** Convención técnica — Tras un reset destructivo de tabla con autoincrement (en este repo: DELETE de `raffle_numbers` durante setup rifa 2026), **el counter de autoincrement NO se resetea por default en SQLite**. Las nuevas filas insertadas siguen desde el último ID + 1. En la rifa 2026 los IDs empiezan en 2001 (después de los 2000 originales) aunque la columna `number` sigue siendo 1-2000. No es bug, pero confunde al hacer queries directas: usar siempre `WHERE number=?` (visible al usuario) y no `WHERE id=?`. Si se necesita reseteo del counter, agregar `DELETE FROM sqlite_sequence WHERE name='raffle_numbers'` antes del INSERT. _(Destino: solo registro)_ _(Origen: descubrimiento al limpiar reservas post-fail durante 4.2)_

### 2026-05-04 — Cloud Scheduler para reemplazar Vercel cron

- **2026-05-04** Herramienta — Para reemplazar `vercel.json` con `crons` en una migración a Cloud Run, el patrón mínimo es: (1) endpoint con auth opcional vía `Authorization: Bearer <secret>` (en este repo ya estaba en `/api/cron/cleanup`), (2) secret en Secret Manager + IAM bind `secretAccessor` al SA del servicio, (3) env var del servicio referenciando el secret (`--update-secrets=CRON_SECRET=cron-secret:latest`), (4) Cloud Scheduler job HTTP en la misma región del Cloud Run con `--headers="Authorization=Bearer ${SECRET}"` leyendo el secret en el mismo comando para no exponerlo. Si el shell del dev tiene `export TURSO_DATABASE_URL=...` (caso real en este proyecto, apuntando a `planificador-docente`), `loadEnv({ path: '.env.local' })` lee el `.env.local` pero el valor del shell gana. Para scripts que MUST tocar la BD definida en `.env.local`, usar `loadEnv({ path: '.env.local', override: true })`. Verificable rápido con: `node --input-type=module -e "import {createClient} from '@libsql/client'; import {config} from 'dotenv'; config({path:'.env.local'}); const c=createClient({url:process.env.TURSO_DATABASE_URL,authToken:process.env.TURSO_AUTH_TOKEN}); console.log(process.env.TURSO_DATABASE_URL); const r=await c.execute(\"SELECT name FROM sqlite_master WHERE type='table'\"); console.log(r.rows);"`. _(Destino: CLAUDE.md como regla — todos los scripts del proyecto deben usar `override: true`)_ _(Origen: bug pre-flight detectado al ejecutar `scripts/setup-rifa-2026.mjs` por primera vez — el script habría reseteado la BD equivocada)_

- **2026-05-04** Técnico — `useEffect` con dep que sea un `useCallback` que a su vez dependa de un state mutado por el propio effect = loop infinito. Patrón concreto en este repo: effect de polling con deps `[loadNumbers, ..., getNumberStatus]`, donde `getNumberStatus = useCallback(..., [numbers])` y `loadNumbers` setea `numbers`. Cada `setNumbers` recrea `getNumberStatus`, que recrea el effect, que llama `loadNumbers`, que setea `numbers`. El estado `loading` queda permanentemente en `true`. **Regla**: los effects de polling/data-fetching deben tener deps mínimas (solo el fetcher estable). Validaciones reactivas que dependen del state ya cargado van en effects separados que NO disparan nuevos fetches. _(Destino: solo registro)_ _(Origen: BUG-009)_

- **2026-05-04** Convención de proceso — Cuando una warning de ESLint `react-hooks/exhaustive-deps` lleva 8+ meses sin atenderse, **probablemente esconde un bug latente** detrás de un early-return o feature flag. La warning de `RifasApp.tsx:629` estaba documentada en LEARNINGS desde 2026-05-01 como "preexistente, no rompe build". Era el síntoma de BUG-009. **Regla**: warnings de hooks deps no son cosmética — son señales de loops potenciales. Si el código entra a la rama del effect, la warning se vuelve bug. _(Destino: solo registro)_ _(Origen: BUG-009 + retrospectiva sobre warnings preexistentes)_

- **2026-05-04** Convención técnica — Constantes globales hardcoded a nivel módulo (ej. `const VENTAS_CERRADAS = true` en `RifasApp.tsx:7`) son una deuda técnica que envejece mal: requieren redeploy para cambiar comportamiento que la BD ya soporta dinámicamente. Para flags de "ventas abiertas/cerradas", derivar siempre de `raffleConfig.isActive` (la BD ya tiene la columna `is_active` en `raffles`). El cambio operativo "abrir/cerrar ventas" debe ser un `UPDATE raffles SET is_active=false`, no un commit + redeploy. **Pendiente**: refactor de `VENTAS_CERRADAS` a derivado de `raffleConfig.isActive`. _(Destino: solo registro)_ _(Origen: descubrimiento al cambiar `VENTAS_CERRADAS=true → false` durante 2.5)_

- **2026-05-04** Técnico — Validación `db-migration-reviewer` antes de ejecutar scripts destructivos contra Turso encontró bug bloqueante real: el orden de DELETE propuesto violaba la FK `event_logs.purchase_id → purchases.id`. La reviewer-agent es invaluable como "second pair of eyes" en operaciones one-shot que tocan datos productivos. Patrón a replicar: **cualquier script que toque la BD productiva debe pasar por `db-migration-reviewer` antes de ejecutarse**, no después. _(Destino: solo registro, ya está implícito en CLAUDE.md → reforzar)_ _(Origen: review pre-flight script setup-rifa-2026.mjs)_

- **2026-05-04** Técnico — Para scripts one-shot que tocan BD productiva, el patrón "2 modos" es robusto: `--backup-only` (default, no destructivo) y `--commit --yes` (requiere doble flag explícito). El `--yes` separado del `--commit` evita ejecuciones accidentales por autocompletado del shell. El backup corre SIEMPRE incluso en `--commit` (es la última red de seguridad), no es redundante con un backup previo manual. _(Destino: solo registro)_ _(Origen: diseño `scripts/setup-rifa-2026.mjs`)_

### 2026-05-02 — Fix BUG-008 (webhook MercadoPago)

- **2026-05-02** Convención técnica — El manifest oficial para verificar HMAC de webhooks IPN de MercadoPago es `id:<data.id>;request-id:<x-request-id>;ts:<ts>;` (separador `;`, trailing `;`, prefijos literales). NO es `<id>.<ts>` ni variantes. El header `x-signature` viene como `ts=...,v1=...` pero el orden NO está garantizado — siempre parsear por nombre. _(Destino: solo registro)_ _(Origen: BUG-008-A diagnóstico)_

- **2026-05-02** Convención técnica — En el handler webhook MP, devolver **5xx en errores transitorios** (BD, MP API timeout, network) habilita la retry policy de MP (3 intentos en ~22 min). Devolver 200 en errores anti-patrón: descarta la red de seguridad. Solo devolver 200 en éxito explícito o ignorables (event types desconocidos). 4xx para validación (firma, body). _(Destino: solo registro)_ _(Origen: BUG-008-F)_

- **2026-05-02** Convención técnica — Funciones de verificación de seguridad (HMAC, JWT, etc.) NUNCA deben tirar excepción ante input malformado. Si `crypto.timingSafeEqual` con buffers de distinta longitud lanza, el catch externo del handler la traga y la lógica falla abierta (devuelve 200). Pre-validar longitudes y envolver en `try/catch` que devuelve `false` por default. _(Destino: solo registro)_ _(Origen: BUG-008-C)_

- **2026-05-02** Técnico — El `ts` del header `x-signature` de MercadoPago viene en **milisegundos** (epoch ms, 13 dígitos), no en segundos. La doc oficial es ambigua — solo se descubre con tráfico real. Heurística para detectar: si `ts > 1e11`, está en ms. Normalizar a segundos solo para validación de ventana de tiempo; mantener el ts original (string) en el manifest HMAC porque ese es el valor que MP firma. _(Destino: solo registro)_ _(Origen: BUG-008-G descubierto en E2E)_

- **2026-05-02** Convención de proceso — Validación E2E con simulación dashboard MP es OBLIGATORIA antes de cerrar fix de webhook. Tests unitarios + curl con firmas forjadas no descubren bugs como 008-G porque dependen del comportamiento real del proveedor. _(Destino: solo registro)_ _(Origen: BUG-008-G)_

- **2026-05-02** Técnico — Idempotencia + locks optimistas en operaciones de BD que un webhook puede invocar múltiples veces: `UPDATE ... WHERE id=? AND paymentStatus='pending'` + `.returning()` + chequeo `length === 0` → throw → catch externo → 503 → MP reintenta cuando estado se estabiliza. Patrón aplicable a cualquier handler que recibe IPNs/notifications con retry policy. _(Destino: solo registro)_ _(Origen: BUG-008-E + concurrency-validator review)_
